## 3.2 数据流分析（Data Flow Analysis）

### 3.2.1 核心概念

数据流分析是一种基于控制流图的编译时分析技术，追踪数据值如何在程序中传播。

**基本框架**：

```
对于程序中的每个程序点 n：

    OUT[n] = GEN[n] ∪ (IN[n] - KILL[n])
    IN[n]  = ∪/∩ { OUT[m] | m 是 n 的前驱 }

其中：
    GEN[n]  — 基本块 n 生成的数据流事实
    KILL[n] — 基本块 n 消除的数据流事实
    IN[n]   — 进入 n 之前成立的事实
    OUT[n]  — 离开 n 之后成立的事实
```

### 3.2.2 经典数据流分析

#### 到达定值分析（Reaching Definitions）

**问题**：变量 x 的某个定义（赋值）能否到达程序的某个点？

```
x = 1           ← def1: x
if (cond) {
    x = 2       ← def2: x
}
print(x)        ← 到达此处的是 def1 还是 def2？还是两者都有可能？

答案：两者都有可能，取决于 cond 的运行时值。
      到达定值分析告诉你"哪些定义有可能到达这里"。
```

**应用**：未初始化变量检测、常量传播、use-def 链构建

#### 活跃变量分析（Live Variable Analysis）

**问题**：变量 x 在某个程序点的值在未来的执行中是否会被使用？

```
x = compute()     ← x 在此被定义
y = 42
print(y)          ← x 在此是否活跃？→ 否，之后不再被使用
return            ← x 在整个下半部分不活跃（死变量）
```

**应用**：寄存器分配、死代码消除、内存优化

#### 可用表达式分析（Available Expressions）

**问题**：表达式 `a + b` 的结果是否已经在之前的执行中被计算过，并且操作数未改变？

```
c = a + b         ← a + b 在此处变为"可用"
...  (a 和 b 未被修改)
d = a + b         ← 可以复用之前的结果！(公共子表达式消除)
```

**应用**：公共子表达式消除、冗余计算消除

### 3.2.3 数据流分析的数学基础

#### 格（Lattice）理论

数据流分析建立在使用格（Lattice）理论的框架上：

```
完备格 (Complete Lattice) 的要求：
1. 偏序集 (Partial Order)：⊑ 关系满足自反性、反对称性、传递性
2. 最小上界：任意子集都有最小上界 (⊔)
3. 最大下界：任意子集都有最大下界 (⊓)
4. 顶元素 (⊤) 和底元素 (⊥)

不动点定理：
在有限高度的格上，单调函数 f 的最小不动点可以通过迭代计算：
    f⁰(⊥) ⊑ f¹(⊥) ⊑ f²(⊥) ⊑ ... ⊑ fⁿ(⊥) = fⁿ⁺¹(⊥)
```

**数据流分析 = 在格上计算不动点**

```
实例：到达定值分析

格：所有定义的幂集，⊑ = ⊇（越精确越"小"）
方向：前向 (Forward)
交汇运算：∪（并集）
初始值：∅（空集，即"无定义到达"）
转移函数：f(IN) = GEN ∪ (IN - KILL)

目标：计算最小不动点（Least Fixed Point）
```

### 3.2.4 数据流分析进阶：IFDS/IDE 框架

| 框架 | 全称 | 能力 | 复杂度 | 工具 |
|------|------|------|--------|------|
| **IFDS** | Interprocedural Finite Distributive Subset | 有限幂集、分配性、过程间分析 | O(ED³) | Soot, FlowDroid |
| **IDE** | Interprocedural Distributive Environment | IFDS + 环境变换 | O(ED³) | Soot, WALA |

**IFDS 能解决的问题**：
```
典型 IFDS 问题：污点分析

定义数据流事实 = { (变量, 污点标签) }
例如: { (x, "tainted"), (y, "clean") }

分配性（Distributive）保证：
f(a ∪ b) = f(a) ∪ f(b)
这意味着每个数据流事实可以独立传播，效率极高。
```

## 3.3 控制流分析（Control Flow Analysis）

### 3.3.1 控制流图（CFG）

控制流图是程序执行路径的图表示：

```
构造规则：
- 节点 = 基本块（连续执行的指令序列，无分支）
- 边 = 可能的控制转移

        ┌──────────────┐
        │   Entry      │
        └──────┬───────┘
               │
        ┌──────▼───────┐
        │  i = 0       │
        │  sum = 0     │  ← 基本块 B1
        └──────┬───────┘
               │
        ┌──────▼───────┐
        │  if i < n    │  ← 基本块 B2
        └──┬────────┬──┘
           │ false  │ true
           │        │
    ┌──────▼───┐  ┌─▼──────────┐
    │ return   │  │ sum += a[i] │ ← B3
    │ sum      │  │ i++         │
    └──────────┘  └──┬──────────┘
                     │
                     └──→ B2 (循环)
```

### 3.3.2 支配树（Dominator Tree）

**支配关系**：节点 d 支配节点 n，当且仅当从入口到 n 的每条路径都经过 d。

```
    Entry
      │
      A    ──── A 支配所有节点
     / \
    B   C  ──── B 支配 D, E   C 支配 F
   / \   \
  D   E   F
          │
         Exit

支配树:
    Entry
      │
      A
     / \
    B   C
   / \   \
  D   E   F
          │
         Exit
```

**应用**：
- 循环识别（自然循环 = 回边 + 支配关系）
- SSA 形式的 φ 节点插入
- 控制依赖分析

### 3.3.3 循环分析

```
循环的识别层次：

1. 自然循环（Natural Loop）
   条件：存在回边 (n → d)，d 支配 n
   循环体 = { n } ∪ { m | m 可达 n 且 d 支配 m }

2. 可归约 vs 不可归约控制流
   可归约：所有循环都是单入口的
   不可归约：存在多入口循环（goto 导致）

3. 循环嵌套树
   外层循环包含内层循环的层次结构
   → 用于循环优化（循环展开、循环交换、循环分块）
```

## 3.4 类型系统与类型检查

### 3.4.1 类型检查作为静态分析

类型检查本质上是一种保守的静态分析——它证明程序满足某种类型安全性。

```
Curry-Howard 同构：
命题 = 类型
证明 = 程序
"程序 P 具有类型 T" = "证明 P 证明了命题 T"

更强的类型系统 = 更强的逻辑 = 能证明更多性质
```

### 3.4.2 类型检查 vs 类型推导

| 特性 | 类型检查 | 类型推导 |
|------|---------|---------|
| **输入** | 带类型注解的程序 | 无/少类型注解的程序 |
| **输出** | 类型错误或通过 | 推导出的类型 |
| **代表语言** | Java, C# | Haskell, OCaml, Rust(局部) |
| **算法** | 类型检查规则 | Hindley-Milner / 统一 |

### 3.4.3 子类型与多态

```
多态类型系统层次：

├── 参数多态 (Parametric Polymorphism)
│   └── 泛型: <T>(x: T) => T
│       基于 System F (多态 lambda 演算)
│
├── 特设多态 (Ad-hoc Polymorphism)
│   ├── 函数重载 (Overloading)
│   └── Type Class (Haskell) / Trait (Rust)
│
├── 子类型多态 (Subtype Polymorphism)
│   ├── 继承 (Java: Dog <: Animal)
│   ├── 结构化子类型 (TypeScript: {x: number, y: number} <: {x: number})
│   └── 协变/逆变 (variance)
│
└── 行多态 (Row Polymorphism)
    └── PureScript, Elm 的记录类型
```

### 3.4.4 协变与逆变（Variance）

```
对于泛型类型 C<T>，T 的变型决定了子类型关系如何传播：

协变 (Covariant):     A <: B  ⇒  C<A> <: C<B>
逆变 (Contravariant): A <: B  ⇒  C<B> <: C<A>   ← 反向！
不变 (Invariant):     A <: B  ⇏  C<A> 与 C<B> 无关系

示例（TypeScript）:
类型  位置      变型    原因
────────────────────────────────────
Array<T>      协变    只读位置（使用 T）
(x: T) => U   逆变    函数参数位置
() => T       协变    函数返回值位置
{ x: T }      协变    T 在只读属性中
Map<K, V>     K 不变  K 同时用于读写
```

```typescript
// 协变示例
class Animal { name: string; }
class Dog extends Animal { breed: string; }

let dogs: Dog[] = [new Dog()];
let animals: Animal[] = dogs;  // ✅ 协变: Dog[] <: Animal[]

// 逆变示例
type Handler<T> = (item: T) => void;
let dogHandler: Handler<Dog> = (d: Dog) => {};
let animalHandler: Handler<Animal> = (a: Animal) => {};

// animalHandler 可以处理任何 Animal，包括 Dog
dogHandler = animalHandler;  // ✅ OK（逆变）
// animalHandler = dogHandler;  // ❌ Error（dogHandler 不能处理 Cat）
```

## 3.5 符号执行（Symbolic Execution）

### 3.5.1 核心原理

用符号值代替具体值执行程序，为每条路径维护路径约束。

```
传统执行:
    x = 5; y = 3; z = x + y;  →  z = 8

符号执行:
    x = α; y = β; z = α + β;
    路径约束: true (无分支)

遇到分支:
    if (x > 0)
        z = x + 1;     路径约束: α > 0, z = α + 1
    else
        z = x - 1;     路径约束: α ≤ 0, z = α - 1
```

### 3.5.2 符号执行树

```
程序:
func test(x, y):
    if (x > 0):
        if (y > 0):
            return 1
        else:
            return 2
    else:
        return 3

符号执行树:
                    σ: {}  PC: true
                         │
              ┌──────────▼──────────┐
              │   if (x > 0)        │
              └──────┬─────────┬────┘
            true     │         │ false
    PC: x > 0        │         │ PC: x ≤ 0
              ┌──────▼──────┐  │  → return 3
              │  if (y > 0) │  │
              └──┬──────┬───┘  │
         true    │      │ false
   PC: x>0∧y>0  │      │ PC: x>0∧y≤0
   → return 1   │      │ → return 2
```

### 3.5.3 路径爆炸与缓解

**符号执行的核心挑战**：路径数量随分支呈指数增长。

| 缓解策略 | 说明 |
|---------|------|
| **路径剪枝** | 提前终止无趣路径 |
| **状态合并** | 将相似状态合并（损失精度） |
| **搜索策略** | BFS/DFS/覆盖率引导/随机搜索 |
| **摘要** | 为函数/循环创建摘要，避免重复探索 |
| **增量求解** | 利用缓存复用 SMT 求解结果 |
| **组合式分析** | 模块化分析，逐步组合结果 |

### 3.5.4 SMT Solver

约束求解是符号执行的核心引擎：

| Solver | 开发者 | 特点 |
|--------|--------|------|
| **Z3** | Microsoft | 最流行，Python/OCaml/C++ API |
| **CVC5** | Stanford/Iowa | 高性能，SMT-COMP 冠军 |
| **Yices 2** | SRI | 轻量高效 |
| **Boolector** | JKU Linz | 位向量优化 |
| **STP** | 学术界 | 先驱，仍广泛使用 |

```
SMT问题示例:

约束:
    x > 0
    y = 2 * x
    y < 5

求解:
    x = 1, y = 2   ← 一个解
    x = 2, y = 4   ← 另一个解

SMT = SAT (布尔可满足) + 理论 (算术、数组、位向量等)
```

## 3.6 抽象解释（Abstract Interpretation）

### 3.6.1 核心思想

在抽象域上计算程序语义的保守近似，通过不动点迭代获取全局性质。

```
具体域 ────→ 抽象域
  (ℤ)        ({-, 0, +})

具体操作:
    3 + (-5) = -2              3 × (-5) = -15

抽象操作:
    γ(3) = +   γ(-5) = -
    + ⊕ - = ⊤ (任意)            + ⊗ - = -  (正×负=负)
```

**抽象解释 = 程序语义的"放大镜"**——以损失精度为代价，换取计算的可行性。

### 3.6.2 符号域示例

| 抽象域 | 表示 | 应用 |
|--------|------|------|
| **符号域 (Sign)** | {⊥, -, 0, +, ⊤} | 除零检测 |
| **区间域 (Interval)** | [a, b] | 数组越界 |
| **同余域 (Congruence)** | a ≡ b (mod m) | 地址对齐 |
| **八边形域 (Octagon)** | ±x ± y ≤ c | 数值不变量 |
| **多面体域 (Polyhedra)** | Σaᵢxᵢ ≤ c | 任意线性不变量 |
| **指针域** | may-point-to 集合 | 别名分析 |

### 3.6.3 工业应用：Astrée

Astrée 是用抽象解释验证空客 A380 飞控软件的成功案例：

```
分析对象: 132,000 行 C 代码
抽象域: 超过 50 个定制的抽象域
效果:
    ✅ 零误报（Zero False Alarm）
    ✅ 证明无运行时错误（除零、溢出、数组越界、非法指针）
    ❌ 需要数小时分析时间 + 专家调优
```

## 3.7 污点分析（Taint Analysis）

### 3.7.1 概念

追踪"不可信"数据（污点源）如何在程序中传播，确保它不会到达"敏感操作点"（污点汇）。

```
污点源 (Sources)            污点汇 (Sinks)
─────────────────         ─────────────────
req.body.username ──→     SQL 查询
req.query.id      ──→     命令执行
document.cookie   ──→     eval()
window.location   ──→     innerHTML
file input        ──→     文件操作

传播:
x = source;     // x 被污染
y = x + 1;      // y 也被污染
z = sanitize(x); // z 是安全的（如果 sanitize 正确）
```

### 3.7.2 显式污点 vs 隐式污点

```javascript
// 显式污点（数据流）
let x = source();   // x: tainted
let y = x;          // y: tainted
let z = y + 1;      // z: tainted

// 隐式污点（控制流）
let x = false;
if (source()) {     // 污点通过控制流影响
    x = true;
}
// x 是否被污染？—— 隐式污点分析认为是
```

## 3.8 程序切片（Program Slicing）

### 3.8.1 定义

**切片准则** `(语句 s, 变量 v)`：提取所有可能影响 s 处 v 的值（后向切片）或被 s 处 v 影响的语句（前向切片）。

```
程序:
1.  read(n);
2.  i = 1;
3.  sum = 0;
4.  product = 1;
5.  while (i <= n) {
6.      sum = sum + i;
7.      product = product * i;
8.      i = i + 1;
9.  }
10. write(sum);
11. write(product);

切片准则: (10, sum)

后向切片:
    1, 2, 3, 5, 6, 8, 10
    (product 相关的 4, 7, 11 被排除)
```

## 3.9 静态分析的形式化定义

### Rice 定理及其含义

**Rice 定理**：程序的任何非平凡语义性质都是不可判定的。

这就意味着——**任何静态分析都无法同时做到完备（Sound）和完备（Complete）**。

```
完备性（Soundness）vs 完整性（Completeness）：

完备 (Sound)  ─ 不漏报 (No False Negatives)
完整 (Complete) ─ 不误报 (No False Positives)

┌────────────┬──────────────┬──────────────┐
│            │ 完备         │ 不完备       │
├────────────┼──────────────┼──────────────┤
│ 完整       │ ✗ 不可能     │ 过度近似     │
│            │  (Rice定理)  │ (Over-approx)│
├────────────┼──────────────┼──────────────┤
│ 不完整     │ 欠近似       │ 启发式       │
│            │ (Under-approx)│ (Heuristic) │
└────────────┴──────────────┴──────────────┘

工业界策略：
- 编译器优化：偏重完备性（必定正确 → 不引入 bug）
- Bug 检测：偏重完整性（减少误报 → 开发者不忽略）
- 安全关键系统：偏重完备性（不能漏掉任何可能的错误）
```

## 本章小结

| 技术 | 核心思路 | 精度 | 扩展性 | 代表工具 |
|------|---------|------|--------|---------|
| AST 模式匹配 | 语法结构匹配 | 低 | 极高 | ESLint, Semgrep |
| 类型检查 | 类型相容性规则 | 中 | 高 | tsc, Mypy |
| 数据流分析 | 格上不动点迭代 | 中 | 高 | CodeQL, Infer |
| 符号执行 | 路径约束 SMT 求解 | 高 | 低 | KLEE, angr |
| 抽象解释 | 抽象域不动点 | 高(可调) | 中 | Astrée, IKOS |
| 模型检查 | 状态空间探索 | 最高 | 最低 | SPIN, TLA+ |

**选择指南**：
- 简单风格/模式检查 → AST 模式匹配
- 类型相关错误 → 类型检查
- 数据流问题（注入、泄漏）→ 数据流/污点分析
- 深度路径问题 → 符号执行
- 安全关键证明 → 抽象解释或模型检查
