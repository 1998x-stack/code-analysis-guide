3.1

数据流分析 (DFA)格理论与不动点

数据流分析是一种基于控制流图的编译时分析技术，追踪数据值如何在程序中传播。其数学基础是格理论（Lattice Theory），分析过程本质上是在格上计算不动点。

dfa-equations.txt · 数据流分析基本方程
// 对于程序中的每个程序点 n： OUT[n] = GEN[n] ∪ (IN[n] - KILL[n]) IN[n] = ∪ 或 ∩ { OUT[m] | m 是 n 的前驱 } // 其中：GEN[n] = 基本块 n 生成的数据流事实 // KILL[n] = 基本块 n 消除的数据流事实 // 不动点定理：在有限高度的格上，单调函数 f 的 // 最小不动点可通过迭代计算： // f⁰(⊥) ⊑ f¹(⊥) ⊑ ... ⊑ fⁿ(⊥) = fⁿ⁺¹(⊥)
分析类型问题应用
到达定值哪些赋值可能到达某程序点？未初始化变量检测、常量传播
活跃变量变量的值在未来是否会被使用？寄存器分配、死代码消除
可用表达式表达式结果是否已被计算？公共子表达式消除
污点分析用户输入会流向哪里？SQL 注入检测、XSS 防护
常量传播哪些变量在运行时总是常量？编译优化、死代码消除
🎯 格（Lattice）理论基础
  • 完备格要求：偏序关系 (⊑)、最小上界 (⊔)、最大下界 (⊓)、顶元素(⊤)和底元素(⊥)
  • 实例（到达定值）：格 = 所有定义的幂集，⊑ = ⊇（越精确越"小"），方向 = 前向
  • IFDS 框架：过程间、有限幂集、分配性分析，复杂度 O(ED³)
3.1 → 3.2 · 关系
数据流分析依赖控制流图（CFG）
数据流分析建立在控制流图之上——CFG 定义了程序点的前后关系，为数据流方程提供了结构基础。

3.3

类型系统与类型检查Curry-Howard 同构

Curry-Howard 同构："命题 = 类型，证明 = 程序"。类型检查本质上是保守的静态分析——它证明程序满足某种类型安全性。更强的类型系统 = 更强的逻辑 = 能证明更多性质。

variance-demo.ts · 协变与逆变
// 协变 (Covariant): A <: B ⇒ C <: C // 逆变 (Contravariant): A <: B ⇒ C <: C ← 反向！ // 不变 (Invariant): A <: B ⇏ 无子类型关系 class Animal { name: string; } class Dog extends Animal { breed: string; } // 协变: Array <: Array ✓ let dogs: Dog[] = [new Dog()]; let animals: Animal[] = dogs; // ✅ OK // 逆变: Handler <: Handler ✓ type Handler<T> = (item: T) => void; let animalHandler: Handler<Animal> = (a) => {}; let dogHandler: Handler<Dog> = animalHandler; // ✅ OK
多态类型系统层次
  • 参数多态：泛型 <T>(x: T) => T，基于 System F
  • 子类型多态：继承 (Dog <: Animal)、结构化子类型 (TypeScript)
  • 特设多态：函数重载、Type Class (Haskell)、Trait (Rust)
3.3 → 3.4 · 更深层分析
类型检查只能证明类型安全——符号执行能探索具体路径
类型系统是保守的（Approximate），而符号执行用符号值模拟每条路径的精确执行——代价是路径爆炸。
3.4

符号执行 (Symbolic Execution)SMT 约束求解

用符号值（α, β, ...）代替具体值执行程序，为每条路径维护路径约束（Path Constraint）。遇到分支时 fork 出两条路径，最后用 SMT Solver 求解约束。

symbolic-execution.txt · 符号执行示例
// 传统执行: 符号执行: x = 5; y = 3; z = x + y; x = α; y = β; z = α + β; // → z = 8 // → 路径约束: true if (x > 0) if (x > 0) z = x + 1; z = x + 1; else else z = x - 1; z = x - 1; // 路径1: α > 0, z = α + 1 // 路径2: α ≤ 0, z = α - 1
SMT Solver开发者特点
Z3Microsoft最流行，Python/OCaml/C++ API
CVC5Stanford/Iowa高性能，SMT-COMP 冠军
Yices 2SRI轻量高效
BoolectorJKU Linz位向量优化
⚠️ 核心挑战：路径爆炸
  • 路径数量随分支呈指数增长
  • 缓解：路径剪枝、状态合并、摘要、增量求解、组合式分析
  • 搜索策略：BFS / DFS / 覆盖率引导 / 随机搜索
3.5

抽象解释Abstract Interpretation

在抽象域上计算程序语义的保守近似，通过不动点迭代获取全局性质。核心思想：用精度换可计算性——具体域 ℤ 映射到抽象域 {'-', '0', '+'}。

抽象域表示应用
符号域 (Sign){⊥, -, 0, +, ⊤}除零检测
区间域 (Interval)[a, b]数组越界
同余域a ≡ b (mod m)地址对齐
八边形域 (Octagon)±x ± y ≤ c数值不变量
多面体域 (Polyhedra)Σaᵢxᵢ ≤ c任意线性不变量
🏭 工业案例：Astrée 验证空客 A380 飞控软件
  • 分析对象：132,000 行 C 代码
  • 抽象域：超过 50 个定制域
  • 效果：零误报，证明无运行时错误（除零、溢出、数组越界、非法指针）
3.6

污点分析与程序切片Taint Analysis & Program Slicing

🔬 污点分析
  • 追踪"不可信"数据如何在程序中传播
  • 污点源：req.body, document.cookie, file input
  • 污点汇：SQL 查询、命令执行、eval()、innerHTML
  • 传播规则：x = source → y = x + 1 → y 也被污染
  • 显式污点（数据流）vs 隐式污点（控制流）
✂️ 程序切片
  • 切片准则：(语句 s, 变量 v)
  • 后向切片：所有可能影响 s 处 v 值的语句
  • 前向切片：所有被 s 处 v 影响的语句
  • 应用：调试定位、影响分析、程序理解
