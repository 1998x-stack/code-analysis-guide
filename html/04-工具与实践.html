4.2

CI/CD 集成最佳实践渐进式引入策略

.github/workflows/quality.yml · GitHub Actions 质量门禁
name: Quality Gate on: pull_request: branches: [main] jobs: fast-checks: strategy: matrix: { check: [lint, format, type-check] } steps: - uses: actions/checkout@v4 - run: npm ci - if: matrix.check == 'lint' run: npx eslint . --max-warnings 0 - if: matrix.check == 'type-check' run: npx tsc --noEmit deep-analysis: needs: fast-checks steps: - uses: actions/checkout@v4 - uses: github/codeql-action/init@v3 - uses: github/codeql-action/analyze@v3 - uses: semgrep/semgrep-action@v1 with: { config: p/security-audit }
🎯 渐进式引入的 4 个阶段
  • 阶段 1 (Week 1-2)：观察——仅报告不阻断，建立基线，修复关键 HIGH 问题
  • 阶段 2 (Week 3-4)：增量阻断——新代码不允许新增 warning，baseline 抑制存量
  • 阶段 3 (Month 2)：全局阻断——不允许 error，warning 呈下降趋势
  • 阶段 4 (Ongoing)：持续优化——根据误报率调整规则，引入业务自定义规则

4.3

实战案例ESLint · Semgrep · CodeQL · ast-grep

案例 1: Type-Aware ESLint
  • 检测 as 类型断言绕过类型检查
  • 禁止 any 类型
  • 强制 optional chaining
  • 检测未处理的 Promise
  • 禁止不安全的成员访问
案例 2: Semgrep 安全规则
  • 检测弱哈希算法 (MD5, SHA-1)
  • 检测硬编码密钥 (password, api_key)
  • 检测不安全的 pickle 反序列化
  • 检测 SQL 注入模式
  • 跨语言一致的安全策略
案例 3: CodeQL 污点追踪
  • 自定义 SQL 注入检测查询
  • 追踪用户输入 → SQL 查询的完整路径
  • 定义清理器（参数化查询、转义函数）
  • 路径问题可视化
案例 4: ast-grep 大规模重构
  • axios → fetch 迁移 (YAML 规则)
  • 批量重命名 API
  • 统一导入路径
  • 自动 fix + 交互式 review

构建自定义分析工具技术选型决策树

decision-tree.txt · 需求 → 工具选择
// 技术选型决策树 需求分析 │ ├── 简单模式匹配？ │ └── ast-grep (YAML 规则) > Semgrep > 手写 │ ├── 涉及类型信息？ │ └── TypeScript Compiler API > Babel + TypeScript │ ├── 需要数据流分析（过程间）？ │ └── CodeQL > Infer > 自研 │ ├── 多语言支持？ │ └── tree-sitter (解析) + 自定义代码 │ ├── 安全漏洞挖掘？ │ └── CodeQL (企业级) > Semgrep Pro > 开源 │ └── 极致性能？ └── Rust (syn + walker) > Go (go/ast) > Node (babel)
tree-sitter 分析器
  • 初始化 Parser，设置语言
  • 解析代码 → S-表达式查询
  • 增量解析：只重新解析修改部分
  • 跨语言统一的 API
TypeScript Compiler API
  • 创建 Program → 获取 TypeChecker
  • 遍历 AST：检测 any 类型、非空断言
  • 类型感知分析：获取完整类型信息
  • AST 转换：安全代码重构
4.5

最佳实践清单规则设计 · 团队采用 · 性能优化

原则说明
低误报率误报率 > 10% 时开发者会忽略所有告警
可操作每条规则必须能指导开发者如何修复
有文档解释"为什么"比"是什么"更重要
可自动修复优先选择能 auto-fix 的规则
上下文感知避免跨文件无上下文的规则
可配置允许项目级别定制阈值和排除模式
⚡ 性能优化策略
  • 缓存：ESLint cache、TS incremental
  • 增量分析：lint-staged，仅分析变更文件
  • 并行化：多 worker 并行执行
  • 选更快的工具：Ruff > Flake8，Oxlint > ESLint
  • 分级运行：快速检查在 PR，慢速在 nightly
🔮 未来趋势
  • AI 增强：ML 模型辅助发现复杂 bug
  • Rust 重写：Ruff, Oxlint, Biome
  • 增量/响应式：rust-analyzer, LSP 演进
  • 供应链安全：SBOM + 依赖分析
  • 统一分析栈：tree-sitter + ast-grep
