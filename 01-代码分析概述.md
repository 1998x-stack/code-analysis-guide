# 第 1 章：代码分析概述

---

## 1.1 什么是代码分析？

**代码分析（Code Analysis）** 是指在不执行程序的情况下（静态分析）或通过执行程序（动态分析）来检查、理解和验证程序行为的一系列技术和方法。

### 核心目标

| 目标 | 说明 | 典型场景 |
|------|------|---------|
| **缺陷发现** | 在代码运行前找出潜在 bug | 空指针、内存泄漏、竞态条件 |
| **代码质量** | 维护代码可读性和一致性 | 命名规范、复杂度控制、死代码清理 |
| **安全审计** | 检测安全漏洞和攻击面 | SQL 注入、XSS、敏感信息泄露 |
| **性能优化** | 识别性能瓶颈 | 热路径分析、内存分配热点 |
| **合规检查** | 确保代码符合行业标准 | MISRA C、OWASP Top 10、GDPR |
| **理解重构** | 帮助开发者理解代码结构 | 依赖关系图、调用链分析、影响面评估 |

## 1.2 代码分析的分类

### 1.2.1 静态分析 vs 动态分析

```
                    ┌─────────────────────────────┐
                    │       代码分析               │
                    └─────────────┬───────────────┘
                                  │
            ┌─────────────────────┼─────────────────────┐
            │                                           │
    ┌───────▼───────┐                         ┌───────▼───────┐
    │   静态分析     │                         │   动态分析     │
    │ (Static)      │                         │ (Dynamic)     │
    └───────┬───────┘                         └───────┬───────┘
            │                                         │
    ┌───────┼───────────┐                   ┌────────┼───────────┐
    │       │           │                   │        │           │
    ▼       ▼           ▼                   ▼        ▼           ▼
  Lint  类型检查   数据流分析            插桩      Fuzzing   性能剖析
```

#### 静态分析（Static Analysis）

**定义**：不运行程序，直接分析源代码、字节码或二进制文件。

**特点**：
- ✅ 覆盖所有代码路径（包括冷路径和异常路径）
- ✅ 无需运行环境，CI/CD 友好
- ✅ 可提前发现缺陷（Shift Left）
- ❌ 存在误报（False Positive）和漏报（False Negative）
- ❌ 无法感知运行时状态（如多态、反射、动态加载）

**核心假设**：静态分析的完备性（Soundness）与精确性（Precision）之间存在根本性权衡。

```
完备性（Soundness）←→ 精确性（Precision）

高完备性 + 低精确性 = 大量误报（ESLint with all rules）
低完备性 + 高精确性 = 可能漏报（简单模式匹配）
平衡点 → 工业级工具追求的目标
```

#### 动态分析（Dynamic Analysis）

**定义**：在程序运行时收集信息，分析实际执行行为。

**特点**：
- ✅ 误报率低（基于实际执行路径）
- ✅ 可以发现运行时特定的问题（内存泄漏、死锁、并发 bug）
- ❌ 只能覆盖被执行的代码路径
- ❌ 需要运行环境和测试用例
- ❌ 性能开销大（特别是插桩分析）

**典型技术**：
- **代码插桩（Instrumentation）**：在程序中插入监控代码
- **模糊测试（Fuzzing）**：自动生成大量输入以触发异常
- **性能剖析（Profiling）**：测量执行时间和资源消耗
- **内存检测**：Valgrind、AddressSanitizer
- **动态污点追踪**：跟踪敏感数据在运行时的流向

### 1.2.2 分析层次

```
源代码 ──→ AST 分析 ──→ IR 分析 ──→ 二进制分析
  │            │           │            │
  ├─ 文本级     ├─ 语法级    ├─ 中间表示级  ├─ 机器码级
  ├─ grep      ├─ ESLint   ├─ LLVM IR   ├─ objdump
  ├─ regex     ├─ Prettier ├─ Clang SA  ├─ Ghidra
  └─ 简单匹配   └─ ast-grep └─ Soufflé   └─ Binary Ninja
```

| 分析层次 | 分析对象 | 信息丰富度 | 典型工具 |
|---------|---------|-----------|---------|
| **词法/文本级** | Token 流、正则匹配 | 低 | grep, sed, Semgrep (部分) |
| **语法级 (AST)** | 抽象语法树 | 中 | ESLint, Prettier, Babel, ast-grep |
| **语义级** | 类型信息、符号表、CFG | 高 | TypeScript Compiler, Clang SA |
| **IR 级** | 中间表示（LLVM IR, Jimple） | 很高 | Clang Static Analyzer, Soot |
| **二进制级** | 汇编、机器码 | 最高（需反编译） | Ghidra, IDA Pro, Binary Ninja |

## 1.3 静态分析的核心能力矩阵

### 1.3.1 基于 AST 的分析

```
源代码: if (x = 1) { ... }
          ↓ 词法分析
Tokens: [if] [(] [x] [=] [1] [)] [{] [...] [}]
          ↓ 语法分析
AST:    IfStatement
        ├── condition: AssignmentExpression
        │   ├── left: Identifier("x")
        │   ├── operator: "="
        │   └── right: NumericLiteral(1)
        └── body: BlockStatement {...}
          ↓ AST 遍历 → 规则检查
结果: ⚠️ 检测到条件语句中使用赋值 (=)，可能误用了比较 (==)
```

**基于 AST 的分析可以做什么？**

| 应用 | 说明 | 工具示例 |
|------|------|---------|
| **Linting** | 检查代码风格和潜在错误 | ESLint, Pylint, Clippy |
| **格式化** | 统一代码格式 | Prettier, Black, gofmt |
| **代码转换** | 语法转换、polyfill、转译 | Babel, SWC, tsc |
| **复杂度分析** | 圈复杂度、认知复杂度 | ESLint complexity rule |
| **依赖分析** | 模块依赖关系、循环依赖检测 | depcheck, madge |
| **重构辅助** | 安全重命名、提取函数 | IDE 重构功能 |

### 1.3.2 基于类型系统的分析

类型系统本身就是一种静态分析——它证明程序的某些性质在所有可能执行中保持不变。

```
类型安全阶梯:

动态类型 ←→ 渐进类型 ←→ 静态类型 ←→ 强静态类型 ←→ 依赖类型
(Python)   (TS/MyPy)   (Java/Go)   (Rust/Haskell)  (Idris/Agda)
                                               ↑
                              类型可以表达的不变量越来越强
```

**类型分析能保证的性质**：
- **JVM/NET 类型安全**：不会将整数当指针使用
- **Rust 所有权**：无数据竞争、无 use-after-free
- **Haskell 纯函数**：无副作用（通过类型）
- **TypeScript 结构化类型**：接口兼容性保证
- **OCaml 穷举检查**：模式匹配覆盖所有 case

### 1.3.3 数据流分析（Data Flow Analysis）

追踪数据在程序中的传播路径，回答"这个值从哪里来，到哪里去？"。

| 分析类型 | 问题 | 应用 |
|---------|------|------|
| **到达定值** | 哪些赋值可能到达某个程序点？ | 未初始化变量检测 |
| **活跃变量** | 某个变量的值在未来是否会被使用？ | 寄存器分配、死代码消除 |
| **可用表达式** | 某个表达式的结果是否已经被计算过？ | 公共子表达式消除 |
| **污点分析** | 用户输入（污点源）会流向哪里？ | SQL 注入检测、XSS 防护 |
| **常量传播** | 哪些变量在运行时总是常量？ | 编译优化、死代码消除 |

### 1.3.4 控制流分析（Control Flow Analysis）

分析程序的所有可能执行路径。

```
控制流图 (CFG) 示例:

        ┌─────┐
        │ 入口 │
        └──┬──┘
           │
    ┌──────▼──────┐
    │ if (x > 0)  │
    └──┬──────┬───┘
       │      │
  true │      │ false
       │      │
    ┌──▼──┐ ┌─▼──┐
    │ a=1 │ │ a=2│
    └──┬──┘ └─┬──┘
       └───┬──┘
           │
    ┌──────▼──────┐
    │  return a   │
    └─────────────┘
```

**控制流分析能检测的问题**：
- 不可达代码（Dead Code）
- 缺少 return 语句
- 无限循环
- 异常处理路径覆盖
- 资源泄漏（某些路径未释放资源）

## 1.4 应用场景全景

### 1.4.1 开发阶段（IDE 集成）

```
编码时 → 实时反馈 → 快速修复

┌──────────────────────────────────────────┐
│  Language Server Protocol (LSP)          │
│  ┌──────────────────────────────────┐   │
│  │ 实时诊断  │ 自动补全 │ 跳转定义  │   │
│  │ 悬停提示  │ 代码重构 │ 格式化    │   │
│  └──────────────────────────────────┘   │
│         ↕ JSON-RPC                      │
│  ┌──────────────────────────────────┐   │
│  │ Language Server                   │   │
│  │ (tsserver, rust-analyzer,        │   │
│  │  gopls, pyright, clangd...)      │   │
│  └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

### 1.4.2 CI/CD 流水线（质量门禁）

```yaml
# 典型的 CI 质量门禁流程
stages:
  - lint          # ESLint / Pylint / Clippy
  - type-check    # tsc --noEmit / mypy / go vet
  - security      # Semgrep / CodeQL / Bandit
  - test          # 单元测试 + 覆盖率
  - build         # 仅当以上全部通过
```

**质量门禁策略**：
| 级别 | 策略 | 阻断条件 |
|------|------|---------|
| **Error** | 阻断合并 | 任何 error 级别问题 |
| **Warning** | 增量检查 | 不允许新增 warning |
| **Info** | 仅报告 | 不阻断，仅可见 |

### 1.4.3 安全审计

| 威胁类型 | 静态分析技术 | 工具 |
|---------|-------------|------|
| SQL 注入 | 污点分析 | CodeQL, Semgrep |
| XSS | 污点分析 + AST 模式 | ESLint security plugins |
| 命令注入 | 数据流分析 | CodeQL, Bandit |
| 路径遍历 | 污点分析 | Semgrep |
| 硬编码密钥 | AST 模式匹配 | TruffleHog, GitGuardian |
| 反序列化漏洞 | AST 模式 + 类型分析 | SpotBugs, Semgrep |

### 1.4.4 大规模代码库管理

Google 的经验（Sadowski et al., 2018）：
- **Tricorder**：Google 的代码审查静态分析平台
- 每天分析 **50,000+** 次代码变更
- 分析结果必须在 **10 分钟内** 返回
- 开发者修复率 > **70%**（因为结果高度相关）
- 关键原则：**低延迟 + 高信噪比 >> 完备性**

## 1.5 静态分析的局限性

| 局限 | 说明 | 缓解措施 |
|------|------|---------|
| **Rice 定理** | 程序的任何非平凡性质都是不可判定的 | 接受不完备性，做近似分析 |
| **误报/漏报** | 完备性与精确性不可兼得 | 用户反馈机制、基线抑制 |
| **路径爆炸** | 路径数量随分支呈指数增长 | 摘要分析、函数摘要 |
| **跨语言** | 多语言项目难以统一分析 | tree-sitter 统一解析 |
| **动态特性** | 反射、动态加载、eval 难以静态分析 | 混合动态分析 |
| **配置开销** | 配置和维护分析规则成本高 | 渐进式引入、自动配置 |

---

## 本章小结

- 代码分析分为 **静态分析**（不运行程序）和 **动态分析**（运行时分析）
- 静态分析的核心权衡：**完备性 vs 精确性**
- AST 是静态分析的基石，提供了代码的结构化表示
- 数据流分析和控制流分析揭示了更深层的程序语义
- 实用主义原则：工业界更看重 **低延迟 + 高信噪比** 而非理论完备性
